Functional vs. Object Oriented Programming

 

"Object-oriented languages are good when you have a fixed set of operations on things, and as your code evolves, you primarily add new things. This can be accomplished by adding new classes which implement existing methods, and the existing classes are left alone.

Functional languages are good when you have a fixed set of things, and as your code evolves, you primarily add new operations on existing things. This can be accomplished by adding new functions which compute with existing data types, and the existing functions are left alone."
-Norman Ramsey

 

Function programming is a programming that involves mostly manipulating data through mathematical functions, like .sort, .inject or .each in Ruby. Then, the programmer can encapsulate ('package') them into methods that accept a set of data, and use it to modify or spit back more data..
Since the method are functions only accept 'static' data and only depend on the nature of the data to properly work, the functional programmer is concerned only with how the data is constructed.

One advantage of using functional programming is precisely the fact that the data is static, and all changes to the program are local. This makes the code much more readable and easier to understand, with less chances of confusion caused by a hidden states or implicit modification of the data.
Additionally, there are no procedural declarations involved. Assigning instance variables is an example of procedural declaration. This makes it easier to follow the flow of the program and to both expect and project results based on the data coming in.

One limitation of using functional programming is that it applies to a smaller scope, hence its use is more limited. With methods as the largest type of encapsulation, any attempt to building more complex or intelligent code involves a large quantity of code, with unavoidable repetitions. As such, the code becomes more difficult to read, and tracking any changes or errors to troubleshoot can be an exhausting task.
Furthermore, since methods accept static data only, it can be difficult to build very complex algorithm using real-world examples. The truth is, the world is mutable and ever-changing, so building programs that mimic events in the real-worlds must accept mutable data.

 

In contrast, Object-Oriented Programming treats objects, not functions, as a basic tool of manipulation. Objects, such as classes, are the largest types of encapsulation, packing variables and methods together in a particular structure. As a result, the OOP programmer will not only be concerned with the construction of the data, but also with how the state of the data is evolving.

note: for a more detailed description and examples of classes, refer to my previous technical post introducing classes.

By encapsulating sets of method and variable into objects and later calling or inheriting these objects, one advantage of using OOP is that is avoids repetition and saves many lines of code. Consequently, this saves programmers time that can be used to think creatively and build more complex coding. Another advantage is that OOPs allows for more complex coding with mutable data based on global changes, not local ones.

On the other hand, while one can argue that encapsulating functions and methods saves lines and is therefore better for readability, I personally disagree. A shorter code does not necessarily mean a more intelligible one. Here is why: with OOP, the mutable state of the data allows the opportunity for more implicit modification to the data. Just like in stocks, it's harder to keep track of what the data actually is at a given time. As a result, the mutable state of the data can make it harder to write test codes, and much easier to make mistake.

Another disadvantage is that OOP is somewhat dependant on functional programming. Functional programming, on the other hand, can exist on its own. Take Haskell, for example. So, what is the purpose of higher-end encapsulation if there is no element to encapsulate to begin with?

In my opinion, the most eloquent of programmers uses a combination of both functional and object-oriented programming to achieve the most elegant code. OOP is most useful for the large, while functional programming is most useful for the small.
